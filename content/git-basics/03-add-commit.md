# 変更を記録する: add / commit

Gitでは、変更したファイルがいきなり履歴に残るわけではありません。まず記録したい変更を選び、その後で履歴として保存します。

- `git add` は、次に記録する変更を選ぶ操作
- `git commit` は、選んだ変更を履歴として保存する操作

この「次に記録する変更を一時的に置く場所」を **ステージング** と呼びます。

この2段階に分かれていることで、関係ある変更だけをきれいにまとめられます。

> まとめ: `add` で記録対象を選び、`commit` で履歴として保存します。

## 変更が記録されるまでの流れ

Gitでは、変更がコミットとして履歴に残るまでに、次のような段階があります。

```txt
作業フォルダ
  ↓ git add
ステージング
  ↓ git commit
コミット履歴
```

作業フォルダは、実際にファイルを編集している場所です。文章を書き換えたり、コードを修正したりすると、まず作業フォルダに変更が生まれます。

ステージングは、次のコミットに入れる変更を置いておく場所です。買い物かごのように、「今回はこれを記録する」と選んだ変更を一時的に集めておくイメージです。

コミット履歴は、ステージングに置いた変更を正式に記録したものです。

> ポイント: ファイルを編集しただけでは履歴に残りません。`git add` でステージングに置き、`git commit` で履歴に残します。

## git add

`git add` は、次のコミットに入れたい変更を選ぶコマンドです。

```bash
git add README.md
```

このコマンドを実行すると、`README.md` の変更がステージングに置かれます。つまり、「次のコミットに入れる候補」として選ばれた状態になります。

複数のファイルをまとめて選ぶこともできます。

```bash
git add .
```

`.` は、現在のフォルダ配下の変更をまとめて対象にする指定です。便利ですが、関係ない変更まで入ることがあるので、実行前に `git status` や `git diff` で確認すると安全です。

たとえば、`README.md` を変更したあとに `git status` を見ると、まだステージングされていない変更として表示されます。

```bash
$ git status
Changes not staged for commit:
  modified:   README.md
```

ここで `git add README.md` を実行すると、次のコミットに入る変更として表示されます。

```bash
$ git add README.md
$ git status
Changes to be committed:
  modified:   README.md
```

このように、`git add` は変更をいきなり履歴に保存する操作ではありません。次のコミットに入れるために、ステージングへ移す操作です。

## git commit

`git commit` は、`git add` で選んだ変更を履歴として保存するコマンドです。

```bash
git commit -m "READMEの説明を更新"
```

`-m` の後ろには、コミットメッセージを書きます。

コミットすると、ステージングに置かれている変更がGitの履歴に残ります。

コミット後に `git status` を見ると、作業フォルダがきれいな状態であれば次のように表示されます。

```bash
$ git status
nothing to commit, working tree clean
```

これは、まだコミットしていない変更が残っていない状態です。

## add と commit が分かれている理由

`add` と `commit` が分かれていることで、記録する変更を自分で選べます。

たとえば、同じ作業中に次の2つの変更をしていたとします。

- READMEの説明を修正した
- デザイン用のCSSも少し変えた

この2つが別の目的なら、別々のコミットにした方があとから読みやすくなります。

この場合、まずREADMEだけをステージングしてコミットし、その後でCSSをステージングして別のコミットにできます。

```bash
git add README.md
git commit -m "READMEの説明を更新"

git add styles.css
git commit -m "ボタンの見た目を調整"
```

もし `add` と `commit` が分かれていなければ、作業中の変更がすべてまとめて履歴に残ってしまい、あとから変更の目的を追いにくくなります。

## よいコミット

よいコミットは、あとから読んだ人が判断しやすい単位になっています。

- 1つの目的にまとまっている
- メッセージを読めば理由がわかる
- レビューしやすい大きさになっている

> コミットは作業ログではなく、チームに残す変更の説明です。

## 理解度チェック

Q1. ステージングの説明として最も近いものはどれですか。

- A. リモートリポジトリに変更を送る場所
- B. GitHubでレビューを受ける画面
- C. 次のコミットに入れる変更を一時的に置いておく場所
- D. 過去のコミットをすべて削除する操作

解説: ステージングは、次のコミットに入れる変更を選んで一時的に置いておく場所です。

Q2. `git add README.md` を実行すると何が起きますか。

- A. `README.md` の変更がステージングに置かれる
- B. `README.md` がリモートに送られる
- C. `README.md` の変更履歴がすべて消える
- D. `README.md` が自動でレビューされる

解説: `git add` は、指定したファイルの変更を次のコミットに入れる候補としてステージングへ移す操作です。

Q3. `git commit -m "READMEの説明を更新"` の説明として最も近いものはどれですか。

- A. 作業フォルダの変更をすべて削除する
- B. リモートリポジトリから最新状態を取り込む
- C. 変更されたファイルを一覧表示する
- D. ステージングに置かれている変更を履歴として保存する

解説: `git commit` は、`git add` でステージングした変更をGitの履歴に残す操作です。

Q4. `add` と `commit` が分かれている理由として最も近いものはどれですか。

- A. GitHubの画面を自動で開くため
- B. 記録する変更を自分で選び、目的ごとにコミットを分けやすくするため
- C. コミットメッセージを書かなくてよくするため
- D. すべての変更を必ず1つのコミットにまとめるため

解説: `add` と `commit` が分かれていることで、関係ある変更だけを選んで、読みやすい単位で履歴に残せます。

答え:

- Q1: C
- Q2: A
- Q3: D
- Q4: B
