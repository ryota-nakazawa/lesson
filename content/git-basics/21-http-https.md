# HTTP / HTTPS

HTTPは、ブラウザやアプリとサーバーがやりとりするときの通信ルールです。

HTTPSは、HTTPの通信を暗号化して安全性を高めたものです。

Webページを開くときも、APIからデータを取得するときも、多くの場合はHTTP / HTTPSの仕組みが使われています。

> まとめ: HTTP / HTTPSは、クライアントとサーバーがリクエストとレスポンスをやりとりするためのルールです。

## リクエストとレスポンス

Webでは、ブラウザやアプリがサーバーへリクエストを送り、サーバーがレスポンスを返します。

```txt
ブラウザ・アプリ
  ↓ リクエスト
サーバー
  ↓ レスポンス
ブラウザ・アプリ
```

たとえば、ページを開く、フォームを送信する、データを取得する、といった操作でHTTP通信が行われます。

APIを呼び出すときも同じです。

```txt
フロントエンド
  ↓ ユーザー一覧をください
APIサーバー
  ↓ ユーザー一覧データを返す
フロントエンド
```

## HTTPメソッド

HTTPリクエストでは、「何をしたいのか」をHTTPメソッドで表すことがあります。

| メソッド | よくある意味 | 例 |
| --- | --- | --- |
| GET | データを取得する | ユーザー一覧を取得する |
| POST | 新しいデータを作成する | フォームを送信する |
| PUT / PATCH | 既存データを更新する | プロフィールを更新する |
| DELETE | データを削除する | 投稿を削除する |

初学者のうちは、`GET` は取得、`POST` は送信・作成でよく見る、と押さえると十分です。

## APIのレスポンスで返るもの

APIのレスポンスには、主に次のような情報が含まれます。

| 含まれるもの | 意味 |
| --- | --- |
| ステータスコード | 成功したか、エラーかを示す番号 |
| ヘッダー | レスポンスに関する補足情報 |
| ボディ | 実際に返ってくるデータ |

たとえば、APIからユーザー情報を取得したとき、次のようなレスポンスが返ることがあります。

```http
HTTP/1.1 200 OK
Content-Type: application/json
```

```json
{
  "id": 1,
  "name": "Taro"
}
```

この例では、`200` がステータスコードで、JSONの部分がレスポンスボディです。

## ステータスコード

HTTPでは、レスポンスにステータスコードが含まれます。

ステータスコードは、レスポンスの結果を3桁の数字で表します。大きく分けると、次の5種類です。

| 分類 | 意味 | 代表例 |
| --- | --- | --- |
| 1xx | 処理中・情報 | `100 Continue` |
| 2xx | 成功 | `200 OK`, `201 Created` |
| 3xx | リダイレクト | `301 Moved Permanently`, `302 Found` |
| 4xx | クライアント側のエラー | `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found` |
| 5xx | サーバー側のエラー | `500 Internal Server Error`, `502 Bad Gateway`, `503 Service Unavailable` |

初学者がまず押さえるべきなのは、`2xx` は成功、`4xx` はリクエスト側の問題、`5xx` はサーバー側の問題、という大まかな見方です。

## よく見るステータスコード

API開発やエラー調査でよく見るステータスコードは次です。

| コード | 意味 | よくある場面 |
| --- | --- | --- |
| `200 OK` | 成功 | データ取得が成功した |
| `201 Created` | 作成成功 | 新しいデータの作成が成功した |
| `400 Bad Request` | リクエストが不正 | 入力値や送信形式が間違っている |
| `401 Unauthorized` | 認証が必要・認証失敗 | ログインしていない、トークンが無効 |
| `403 Forbidden` | 権限がない | ログイン済みだが、その操作は許可されていない |
| `404 Not Found` | 見つからない | URLや対象データが存在しない |
| `500 Internal Server Error` | サーバー内部エラー | サーバー側の処理で例外が起きた |
| `503 Service Unavailable` | 一時的に利用不可 | メンテナンス中、負荷が高い |

ステータスコードを見ると、どこに問題がありそうかを絞りやすくなります。

```txt
200番台: 成功している
400番台: リクエストや権限を確認する
500番台: サーバー側の処理やログを確認する
```

## HTTPSが重要な理由

HTTPSでは通信が暗号化されます。

HTTPのままだと、通信内容を第三者に見られたり、改ざんされたりするリスクがあります。

HTTPSを使うと、ブラウザとサーバーの間の通信を安全にしやすくなります。

ログイン情報、個人情報、決済情報を扱うWebサービスでは、HTTPSが前提になります。

| 通信 | 特徴 |
| --- | --- |
| HTTP | 暗号化されていない通信 |
| HTTPS | 暗号化された通信 |

本番公開するWebサイトやAPIでは、基本的にHTTPSを使うと考えておくとよいです。

## まず押さえること

最初に押さえるべきことは次の4つです。

- HTTP / HTTPSは、クライアントとサーバーの通信ルール
- リクエストを送り、レスポンスを受け取る
- APIレスポンスにはステータスコードが含まれる
- HTTPSは通信を暗号化するため、本番では基本的に重要

> HTTP / HTTPSを理解すると、API通信やエラー調査の会話がしやすくなります。

## 理解度チェック

Q1. HTTP / HTTPSの説明として最も近いものはどれですか。

- A. Gitのブランチを管理するための仕組み
- B. クライアントとサーバーがリクエストとレスポンスをやりとりするための通信ルール
- C. パッケージのバージョンだけを固定する仕組み
- D. データベースの表を自動で作るコマンド

解説: HTTP / HTTPSは、ブラウザやアプリとサーバーが通信するときのルールです。

Q2. APIレスポンスに含まれるステータスコードの役割として最も近いものはどれですか。

- A. 成功したか、エラーかなど、レスポンスの結果を示す
- B. GitHubのユーザー名を変更する
- C. HTMLの文字色を指定する
- D. ローカルPCのファイル名を自動で変える

解説: ステータスコードを見ると、API呼び出しが成功したのか、どの種類のエラーなのかを判断しやすくなります。

Q3. `404 Not Found` の意味として最も近いものはどれですか。

- A. サーバー内部で例外が起きた
- B. 認証が必要でログインしていない
- C. URLや対象データが見つからない
- D. 新しいデータの作成が成功した

解説: `404 Not Found` は、アクセスしたURLや対象データが見つからないときによく返ります。

Q4. 500番台のステータスコードを見たときの大まかな見方として最も近いものはどれですか。

- A. 必ず入力フォームの見た目だけが原因
- B. 常に成功している
- C. DNSの名前解決だけが原因
- D. サーバー側の処理やログを確認する

解説: 500番台はサーバー側のエラーを示すため、サーバー側の処理やログを確認するきっかけになります。

Q5. HTTPSが重要な理由として最も近いものはどれですか。

- A. 通信を暗号化して安全性を高めるため
- B. ポート番号を必ず3000に固定するため
- C. Gitのコミットを自動で作るため
- D. APIレスポンスを必ずCSVに変換するため

解説: HTTPSはHTTPの通信を暗号化し、ログイン情報や個人情報などを安全に扱いやすくします。

答え:

- Q1: B
- Q2: A
- Q3: C
- Q4: D
- Q5: A
