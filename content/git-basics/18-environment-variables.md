# 環境変数って何？

環境変数は、アプリケーションの外側から渡す設定値です。

たとえば、APIキー、データベースの接続先、外部サービスのURLなどを環境変数として扱うことがあります。

アプリのコードに直接値を書き込むのではなく、外側から値を渡して、アプリの中でその値を読み取ります。

```txt
環境変数として値を定義する
  ↓
アプリが起動時に値を読み取る
  ↓
API接続先や秘密情報として使う
```

> まとめ: 環境変数は、環境ごとに変わる設定をアプリの外から渡す仕組みです。

## なぜ環境変数を使うのか

ローカル環境と本番環境では、使う設定が違うことがあります。

```txt
ローカル: 開発用DBにつなぐ
本番: 本番用DBにつなぐ
```

このような値をコードに直接書くと、環境ごとの差し替えが難しくなります。

たとえば、APIの接続先をコードに直接書くと、sandbox環境と本番環境でコードを書き換える必要が出てしまいます。

```js
// あまりよくない例: 接続先をコードに直接書いている
const apiUrl = "https://api.example.com";
```

環境変数を使うと、コード側は同じ名前を読むだけにできます。

```js
// 環境変数から接続先を読む
const apiUrl = process.env.API_URL;
```

このようにしておくと、環境ごとに `API_URL` の値を変えるだけで接続先を切り替えられます。

## .envファイル

開発では、`.env` というファイルに環境変数を書くことがあります。

```txt
API_URL=https://api.example.com
DATABASE_URL=postgres://...
```

環境変数は、基本的に次のように「名前=値」の形で定義します。

```txt
変数名=値
```

たとえば、ローカル開発では次のように書くことがあります。

```txt
API_URL=https://sandbox-api.example.com
APP_ENV=local
```

この例では、`API_URL` という名前でAPIの接続先を定義し、`APP_ENV` という名前で今の環境を表しています。

ただし、APIキーやパスワードなどの秘密情報を含む場合は、Gitにコミットしないように注意します。

## アプリからどう呼び出すか

環境変数は、アプリの中から「変数名」を指定して読み取ります。

Node.jsでは、`process.env` から環境変数を読むことがよくあります。

```js
const apiUrl = process.env.API_URL;

console.log(apiUrl);
```

ただし、`.env` に書いた値がどのように読み込まれるかは、使っているフレームワークや実行方法によって変わります。

ViteやNext.jsのように `.env` を読み込む仕組みが用意されているものもあれば、Node.jsのシンプルなスクリプトでは別途読み込み設定が必要になることもあります。

ここではまず、「アプリ側では環境変数の名前を指定して値を読む」と押さえておけば十分です。

`.env` に次のように書かれていれば、

```txt
API_URL=https://sandbox-api.example.com
```

アプリ側では `process.env.API_URL` として値を取り出します。

```txt
process.env.API_URL
  ↓
https://sandbox-api.example.com
```

実際のアプリでは、次のようにAPIの接続先として使うことがあります。

```js
const apiUrl = process.env.API_URL;

fetch(`${apiUrl}/users`);
```

この例では、`API_URL` が `https://sandbox-api.example.com` なら、次のURLにアクセスします。

```txt
https://sandbox-api.example.com/users
```

## フロントエンドでは名前のルールがあることもある

フロントエンドの開発ツールでは、環境変数の名前にルールがあることがあります。

たとえば、Viteではフロントエンド側で使う環境変数に `VITE_` を付けます。

```txt
VITE_API_URL=https://sandbox-api.example.com
```

アプリ側では、次のように読み取ります。

```js
const apiUrl = import.meta.env.VITE_API_URL;
```

このように、使っているフレームワークやツールによって読み取り方が変わることがあります。

| 場面 | 定義例 | 読み取り例 |
| --- | --- | --- |
| Node.jsのサーバー側 | `API_URL=https://...` | `process.env.API_URL` |
| Viteのフロントエンド側 | `VITE_API_URL=https://...` | `import.meta.env.VITE_API_URL` |

初学者のうちは、「環境変数は外側で定義し、アプリの中で名前を指定して読む」と押さえれば十分です。

## 環境ごとに値を変える

同じ `API_URL` でも、環境によって値を変えることがあります。

| 環境 | `API_URL` の例 | 意味 |
| --- | --- | --- |
| ローカル環境 | `http://localhost:3000` | 自分のPC上のAPIにつなぐ |
| sandbox環境 | `https://sandbox-api.example.com` | テスト用APIにつなぐ |
| 本番環境 | `https://api.example.com` | 本番APIにつなぐ |

アプリのコードは同じでも、環境変数の値を変えることで、接続先を切り替えられます。

```txt
同じコード
  ↓
環境変数の値だけ変える
  ↓
ローカル / sandbox / 本番を切り替える
```

## Gitに入れてよいものと入れてはいけないもの

環境変数には、外部に見せてはいけない値が含まれることがあります。

| 値の種類 | Gitに入れるか | 理由 |
| --- | --- | --- |
| APIキー | 入れない | 外部から悪用される可能性がある |
| DBのパスワード | 入れない | データベースに不正アクセスされる可能性がある |
| 公開してよいAPIのURL | 場合による | URLだけなら問題ないこともある |
| `.env.example` | 入れてよいことが多い | 必要な変数名の見本として使える |

実務では、`.env` はGitに入れず、代わりに `.env.example` を用意することがあります。

```txt
.env
  ↓
実際の秘密情報を書く。Gitには入れない。

.env.example
  ↓
必要な変数名の見本を書く。Gitに入れることがある。
```

`.env.example` には、実際の秘密情報ではなく、サンプル値を書きます。

```txt
API_URL=https://example.com
DATABASE_URL=your_database_url_here
```

## FDE人材が押さえること

- 環境変数は環境ごとの設定値
- APIキーや接続先URLでよく使う
- 秘密情報はGitに入れない
- 本番環境では管理画面やCI/CDで設定することがある
- アプリ側では、変数名を指定して値を読み取る

> 環境変数は、アプリを安全に環境ごとへ切り替えるための重要な仕組みです。

## 理解度チェック

Q1. 環境変数の説明として最も近いものはどれですか。

- A. アプリの外側から渡す設定値
- B. Gitのコミット履歴を保存する場所
- C. HTMLの見出しだけを変更する機能
- D. データベースの表を自動作成する仕組み

解説: 環境変数は、APIの接続先や秘密情報など、環境ごとに変わる設定をアプリの外側から渡すために使います。

Q2. Node.jsで `API_URL` という環境変数を読み取る例として最も近いものはどれですか。

- A. `git.env.API_URL`
- B. `process.env.API_URL`
- C. `document.API_URL`
- D. `npm.API_URL`

解説: Node.jsでは、`process.env.API_URL` のように `process.env` から環境変数を読み取ることがよくあります。

Q3. Viteのフロントエンド側で使う環境変数として適切な名前はどれですか。

- A. `API_URL`
- B. `NODE_API_URL`
- C. `VITE_API_URL`
- D. `GIT_API_URL`

解説: Viteでは、フロントエンド側で使う環境変数に `VITE_` を付けるルールがあります。

Q4. `.env` と `.env.example` の扱いとして最も近いものはどれですか。

- A. `.env` も `.env.example` も必ず本番のAPIキーをそのまま書いてGitに入れる
- B. `.env.example` はアプリの実行ログだけを保存するファイル
- C. `.env` はGitのブランチ名を自動で変えるためのファイル
- D. `.env` には秘密情報を書くことがあるのでGitに入れず、`.env.example` には必要な変数名の見本を書くことがある

解説: `.env` には実際の秘密情報が入ることがあるためGitに入れず、`.env.example` は必要な変数名を共有する見本として使うことがあります。

Q5. sandbox環境では動くのに本番環境ではAPI接続に失敗しました。まず確認することとして最も近いものはどれですか。

- A. 本番用の `API_URL`、APIキー、権限などの環境変数が正しく設定されているか
- B. HTMLの見出しタグがすべて大文字になっているか
- C. Gitのコミットメッセージが日本語かどうか
- D. `.env.example` に本番の秘密情報を直接書いているか

解説: 環境ごとに接続先やAPIキーが違うため、本番環境で失敗する場合は本番用の環境変数や権限を確認することが重要です。

答え:

- Q1: A
- Q2: B
- Q3: C
- Q4: D
- Q5: A
