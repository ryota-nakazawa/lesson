# ドメイン / DNS

Webサイトにアクセスするとき、私たちは `example.com` のような文字列をブラウザに入力します。

この `example.com` のような名前を **ドメイン** と呼びます。

ただし、コンピューター同士の通信では、最終的に **IPアドレス** のような接続先情報が使われます。

DNSは、ドメインをIPアドレスなどの接続先へ変換する仕組みです。

このように、ドメイン名からIPアドレスなどの接続先を調べることを **名前解決** と呼びます。

> まとめ: ドメインは人間が読みやすい住所、DNSはその住所を接続先へ変換する仕組みです。

## ドメインとは

ドメインは、Webサイトやサービスにアクセスするための人間向けの名前です。

たとえば、次のようなものがドメインです。

| 例 | 説明 |
| --- | --- |
| `example.com` | Webサイトでよく見るドメイン |
| `github.com` | GitHubのドメイン |
| `google.com` | Googleのドメイン |

人間にとっては、数字の羅列よりも `example.com` のような名前の方が覚えやすく、扱いやすいです。

## IPアドレスとは

IPアドレスは、ネットワーク上の接続先を識別するための番号のようなものです。

たとえば、次のような形式があります。

```txt
192.0.2.10
```

実際の通信では、ブラウザがドメイン名だけを見ているわけではありません。裏側では、そのドメインがどのIPアドレスに対応しているかを調べて、接続先を見つけています。

```txt
人間: example.com にアクセスしたい
  ↓
コンピューター: 対応するIPアドレスを調べる
  ↓
接続先のサーバーへアクセスする
```

初学者のうちは、ドメインは「人間が覚えやすい名前」、IPアドレスは「コンピューターが接続先を見つけるための番号」と考えると分かりやすいです。

## なぜDNSが必要か

コンピューター同士は、最終的にはIPアドレスのような接続先情報を使って通信します。

しかし、人間が毎回IPアドレスを覚えるのは大変です。

そこで、`example.com` のようなドメインを使います。

```txt
example.com
  ↓ DNSで名前解決
192.0.2.10
  ↓
接続先のサーバー
```

DNSは、電話帳のような役割だと考えると分かりやすいです。

| 人間が使う名前 | DNSが調べる接続先 |
| --- | --- |
| `example.com` | `192.0.2.10` |
| `app.example.com` | `192.0.2.20` |

実際にはもっと複雑ですが、まずは「DNSはドメインを接続先へ名前解決する仕組み」と押さえれば十分です。

## サブドメインとは

サブドメインは、ドメインの前に名前を付けて用途を分けるものです。

たとえば、`example.com` というドメインがある場合、次のような名前を作れます。

| 名前 | 用途の例 |
| --- | --- |
| `example.com` | 会社サイトやサービス本体 |
| `app.example.com` | アプリの画面 |
| `api.example.com` | APIサーバー |
| `docs.example.com` | ドキュメント |

同じ `example.com` の配下でも、用途ごとに接続先を分けることがあります。

## DNSレコードとは

DNSレコードは、ドメインをどこへ向けるかを決める設定です。

代表的なものに、次のようなレコードがあります。

| レコード | ざっくりした役割 |
| --- | --- |
| Aレコード | ドメインをIPv4アドレスへ向ける |
| CNAMEレコード | あるドメインを別のドメイン名へ向ける |
| MXレコード | メールを受け取るサーバーを指定する |

たとえば、`example.com` を `192.0.2.10` に向けたい場合、Aレコードを設定することがあります。

```txt
example.com
  ↓ Aレコード
192.0.2.10
```

一方で、`www.example.com` を別のサービスのドメインへ向けるときは、CNAMEレコードを使うことがあります。

```txt
www.example.com
  ↓ CNAMEレコード
service.example.net
```

## Webサイト公開で何が起きるか

Webサイトを公開するときは、アプリをデプロイするだけでは足りないことがあります。

利用者がURLでアクセスできるようにするには、ドメインが正しい接続先を向いている必要があります。

```txt
アプリをデプロイする
  ↓
サーバーやホスティング先が用意される
  ↓
DNSでドメインを接続先へ向ける
  ↓
example.com でアクセスできる
```

DNSの設定を間違えると、アプリ自体は動いていても、URLからアクセスできないことがあります。

## よく見る言葉の整理

| 言葉 | 意味 |
| --- | --- |
| ドメイン | `example.com` のような人間が読みやすい名前 |
| IPアドレス | 接続先を識別するための番号 |
| DNS | ドメインをIPアドレスなどの接続先へ変換する仕組み |
| 名前解決 | ドメイン名からIPアドレスなどの接続先を調べること |
| サブドメイン | `app.example.com` のように用途を分ける名前 |
| DNSレコード | ドメインをどこへ向けるかの設定 |

## トラブル時に見るポイント

Webサイトを公開するときは、アプリをデプロイするだけでなく、ドメインが正しい接続先を向いている必要があります。

特に、次のような状況ではDNSが関係していることがあります。

- デプロイしたのに独自ドメインでアクセスできない
- `www.example.com` は開けるが `example.com` は開けない
- `api.example.com` にアクセスすると別のサービスへつながる
- ドメイン設定を変えたのにすぐ反映されない

DNSレコードをすべて暗記する必要はありません。ただし、「ドメインはDNS設定によって接続先へ向けられている」という考え方を押さえると、トラブル時に原因を切り分けやすくなります。

> ドメインとDNSが分かると、「URLでアクセスできるようにする」流れが理解しやすくなります。

## 理解度チェック

Q1. ドメインの説明として最も近いものはどれですか。

- A. Webサイトやサービスにアクセスするための人間向けの名前
- B. Gitのコミット履歴を保存する場所
- C. パッケージをインストールするためのコマンド
- D. データベースのテーブル名だけを管理する仕組み

解説: ドメインは、`example.com` のように人間が読みやすい名前として使われます。

Q2. 名前解決の説明として最も近いものはどれですか。

- A. サーバーのCPU使用率を下げること
- B. ドメイン名からIPアドレスなどの接続先を調べること
- C. Gitのブランチ名を変更すること
- D. HTMLの見出しを自動で翻訳すること

解説: 名前解決は、`example.com` のようなドメイン名から、接続先となるIPアドレスなどを調べることです。

Q3. Aレコードの役割として最も近いものはどれですか。

- A. メールを受け取るサーバーを指定する
- B. あるドメインを別のドメイン名へ向ける
- C. ドメインをIPv4アドレスへ向ける
- D. JavaScriptの依存関係を管理する

解説: Aレコードは、ドメインをIPv4アドレスへ向けるためのDNSレコードです。

Q4. アプリをデプロイしたのに独自ドメインでアクセスできない場合、確認する観点として最も近いものはどれですか。

- A. `node_modules/` をGitに入れているかだけを確認する
- B. Pull Requestのタイトルが英語か確認する
- C. ローカル環境のポート番号だけを必ず変更する
- D. DNSレコードが正しい接続先を向いているか確認する

解説: アプリ自体が動いていても、DNSレコードが正しい接続先を向いていないと、独自ドメインからアクセスできないことがあります。

答え:

- Q1: A
- Q2: B
- Q3: C
- Q4: D
