# RAGとは何か

RAGは、Retrieval-Augmented Generationの略です。

日本語では「検索拡張生成」と呼ばれることがあります。簡単に言うと、外部の資料を検索して、その内容をLLMに渡して回答させる仕組みです。

> まとめ: RAGは、LLMだけで答えさせるのではなく、必要な資料を検索してから回答を作る仕組みです。

## なぜRAGが必要か

LLMは便利ですが、次のような弱点があります。

| 弱点 | 例 |
| --- | --- |
| 最新情報に弱い | 新しい社内ルールや直近の仕様変更を知らない |
| 社内情報を知らない | 社内ドキュメントやプロジェクト固有の情報を持っていない |
| 誤情報を出す | もっともらしいが間違った回答をすることがある |

そこで、回答に必要な資料を先に探し、その資料を根拠としてLLMに渡します。

## RAGの流れ

RAGの基本的な流れは次のようになります。

```txt
ユーザーの質問
  ↓
関連する資料を検索
  ↓
見つかった資料をLLMに渡す
  ↓
資料をもとに回答を生成
```

たとえば、社内FAQに対して質問するAIを作る場合、LLMに丸投げするのではなく、まず社内FAQの中から関係するページを探します。

そのうえで、見つかった内容をLLMに渡して回答を作らせます。

## 問い合わせBotでの使い方

RAGを使うと、社内FAQやマニュアルをもとに回答する問い合わせBotを作りやすくなります。

たとえば、次のようなBotです。

| 利用場面 | 参照する資料 |
| --- | --- |
| 社内ルールBot | 就業規則、申請手順、社内FAQ |
| カスタマーサポートBot | 製品マニュアル、FAQ、過去の問い合わせ |
| 開発チーム向けBot | 設計書、API仕様、運用手順 |

ユーザーが質問すると、Botはまず関連しそうな資料を探します。

その後、見つかった資料をLLMに渡し、資料を根拠にして回答します。

```txt
ユーザー: 経費申請の締切はいつですか？
  ↓
社内FAQから「経費申請」の資料を検索
  ↓
該当するルールをLLMに渡す
  ↓
Bot: 締切は毎月25日です。詳細は...
```

このように、RAGは「LLMに全部覚えさせる」のではなく、「必要な資料を探してから答えさせる」仕組みです。

## ベクトル化と埋め込み検索

RAGでは、埋め込みを使った意味検索がよく使われます。

埋め込みとは、文章の意味を数値のベクトルとして表したものです。

文章をベクトルに変換することを、ここではベクトル化と呼びます。

RAGでは、質問文と資料をベクトル化し、意味が近い資料を探します。

```txt
質問文
  ↓ 埋め込み
資料
  ↓ 埋め込み
意味が近い資料を検索
```

キーワードが完全一致しなくても、意味が近い資料を探せる点が便利です。

たとえば、ユーザーが「休みを取りたい」と質問した場合でも、資料側に「有給休暇の申請方法」と書かれていれば、意味が近い情報として見つけられる可能性があります。

## RAGの準備と実行

RAGでは、質問が来てからすべての資料を毎回読み込むわけではありません。

多くの場合、事前に資料を分割して、ベクトル化しておきます。

```txt
事前準備
社内FAQやマニュアル
  ↓ 小さな文章に分割
  ↓ 埋め込みモデルでベクトル化
  ↓ ベクトルDBや検索用DBに保存
```

ユーザーから質問が来たら、質問文もベクトル化します。

そして、保存してある資料ベクトルの中から、質問ベクトルに近いものを探します。

```txt
実行時
ユーザーの質問
  ↓ ベクトル化
近い資料を検索
  ↓
資料をLLMに渡す
  ↓
回答を生成
```

この仕組みによって、LLMは社内資料や最新資料を参照しながら回答できます。

> ベクトルDBは、ベクトル化された文章を保存し、意味が近いものを探しやすくするためのDBです。まずは「意味検索をするための保存場所」と考えると分かりやすいです。

## RAGで注意すること

RAGを使えば必ず正しくなるわけではありません。

次の点に注意が必要です。

| 観点 | 注意点 |
| --- | --- |
| 検索品質 | 関係ない資料を渡すと回答もずれる |
| 資料の鮮度 | 古い資料をもとにすると古い回答になる |
| 権限 | 見せてはいけない資料を検索対象にしない |
| 回答品質 | LLMが資料を読み違えることがある |

特に社内情報を扱う場合は、誰がどの資料にアクセスできるかを設計することが重要です。

## まず押さえること

- RAGは、検索した資料をLLMに渡して回答させる仕組み
- RAGを使うと、社内FAQやマニュアルをもとにした問い合わせBotを作りやすい
- 最新情報や社内情報を扱うときに役立つ
- 文章をベクトル化し、埋め込み検索で意味が近い資料を探す
- 事前に資料を分割・ベクトル化して保存し、質問時に近い資料を検索する
- 検索品質、資料の鮮度、権限管理が重要

> RAGは、LLMに知識を丸暗記させるのではなく、必要な資料を参照させながら回答させる考え方です。

## 理解度チェック

Q1. RAGの説明として最も適切なものはどれですか。

- A. LLMに社内資料をすべて暗記させ、検索を使わずに回答させる仕組み
- B. 外部資料を検索し、その内容をLLMに渡して回答を生成する仕組み
- C. 質問文を無視して、毎回同じテンプレートを返す仕組み
- D. LLMの回答が必ず正しいことを保証する検証機構

解説: RAGは、検索した資料をLLMに渡して回答を生成する仕組みです。

Q2. 問い合わせBotにRAGを使う理由として最も適切なものはどれですか。

- A. FAQやマニュアルの内容を検索し、その資料を根拠に回答しやすくするため
- B. Botが資料を参照せず、必ず記憶だけで回答できるようにするため
- C. ユーザーの質問を保存せず、すべての資料を削除するため
- D. Botの回答から権限管理を完全になくすため

解説: RAGは、LLMがもともと知らない社内情報や最新情報を参照させたいときに役立ちます。

Q3. RAGでベクトル化や埋め込み検索を使う理由として最も適切なものはどれですか。

- A. 文章の意味を数値ベクトルにして、質問と意味が近い資料を探しやすくするため
- B. 資料の意味を失わせ、文字数だけで検索するため
- C. LLMの学習済みパラメータを毎回直接書き換えるため
- D. 検索結果に関係なく、ランダムな資料をLLMに渡すため

解説: 埋め込み検索を使うと、キーワード完全一致だけでなく意味の近さで資料を探せます。

Q4. RAGの事前準備と実行時の流れとして最も適切なものはどれですか。

- A. 質問が来るたびに、すべての資料を削除してからLLMに渡す
- B. 資料を検索せず、LLMの学習済み知識だけで回答を固定する
- C. 資料をベクトル化せず、常にファイル名だけで回答を作る
- D. 事前に資料を分割・ベクトル化して保存し、質問時に近い資料を検索してLLMに渡す

解説: RAGでは、事前に資料を分割・ベクトル化して保存し、質問時に意味が近い資料を検索してLLMに渡す構成がよく使われます。

Q5. RAGで注意すべきこととして適切なものはどれですか。

- A. 古い資料でも必ず最新回答になる
- B. 関係ない資料を渡しても回答品質に影響しない
- C. 権限管理や資料の鮮度を確認する必要がある
- D. LLMの回答確認は一切不要になる

解説: RAGでは、検索対象の権限、資料の鮮度、検索品質が回答に大きく影響します。

答え:

- Q1: B
- Q2: A
- Q3: A
- Q4: D
- Q5: C
