# Agent skillsって何？

Agent skillsは、AIエージェントに特定の作業手順や判断基準を渡すための仕組みです。

毎回同じ説明をしなくても、スキルとしてまとめておくことで、AIがその作業に合った進め方をしやすくなります。

> まとめ: Agent skillsは、AIに特定作業の進め方を渡すための再利用できる手順書です。

## なぜスキルが必要か

AIは汎用的に会話できますが、プロジェクト固有の作法までは最初から知りません。

たとえば、次のようなことは毎回説明すると手間がかかります。

| 作業 | 毎回説明が必要になりやすいこと |
| --- | --- |
| 学習コンテンツ作成 | ファイル配置、検証コマンド、チェックテストの形式 |
| レビュー | どの観点で品質を見るか、優先度をどう付けるか |
| ドキュメント作成 | 見出し構成、文体、表や図の使い方 |
| 開発作業 | テスト手順、ビルド手順、リポジトリ固有の注意点 |

こうした作法をスキルにしておくと、AIが作業前にその手順を読み、同じ基準で進めやすくなります。

```txt
毎回プロンプトで説明する
  ↓
よく使う手順をスキル化する
  ↓
同じ作業を安定して繰り返しやすくなる
```

## スキルには何を書くのか

スキルには、主に次のような情報を書きます。

| 書く内容 | 例 |
| --- | --- |
| いつ使うか | チェックテストを作るとき、学習コンテンツをレビューするとき |
| 作業手順 | 先にMarkdownを読む、最後に検証コマンドを実行する |
| 判断基準 | 初学者にわかりやすいか、正解位置が偏っていないか |
| 参照するファイル | `course.json`、対象Markdown、画像ファイル |
| 完了前の確認 | バンドル再生成、表示崩れ、テスト結果 |

このプロジェクトでも、学習コンテンツ管理、チェックテスト作成、学習コンテンツレビューの流れをスキルとして管理しています。

## 社内ナレッジをスキル化する意味

Agent skillsは、個人の作業メモだけではありません。

チームや会社の中には、レビュー観点、ドキュメントの書き方、リリース前の確認手順、顧客対応の方針など、明文化されていない暗黙知が多くあります。

こうした社内特有のナレッジをスキルとして言語化しておくと、AIエージェントがその作法を参照しながら作業しやすくなります。

| 社内ナレッジ | スキル化するとよい理由 |
| --- | --- |
| レビュー観点 | 指摘の粒度や優先度をそろえやすい |
| ドキュメント作法 | 見出し構成や文体をそろえやすい |
| リリース手順 | 確認漏れを減らしやすい |
| 顧客対応方針 | 回答前に守るべき基準を渡しやすい |
| 社内ツールの使い方 | 毎回説明しなくても手順を参照できる |

また、スキルはCodexやClaude CodeのようなAIエージェントで使えるため、一度整備したナレッジを複数の作業場面で再利用しやすくなります。

つまり、スキル化は「AIに手順を教えること」であると同時に、「社内ナレッジをAIが使える形に整えること」でもあります。

AIをうまく使うには、プロンプトをその場で工夫するだけでなく、組織の作法や判断基準をAIが参照できる形にしておくことが重要です。

## 基本的なディレクトリ構成

Agent skillsは、基本的に1つのスキルを1つのディレクトリとして管理します。

中心になるのは `SKILL.md` です。ここに、スキルの説明、使う場面、作業手順、注意点などを書きます。

```txt
my-skill/
  SKILL.md
  agents/
    openai.yaml
  scripts/
    helper.js
  assets/
    template.md
  references/
    guide.md
```

よく使われる構成は次の通りです。

| 場所 | 役割 |
| --- | --- |
| `SKILL.md` | スキルの本体。いつ使うか、どう進めるかを書く |
| `agents/openai.yaml` | OpenAI系エージェント向けの設定やメタ情報を置く |
| `scripts/` | 固定処理や繰り返し処理をスクリプトとして置く |
| `assets/` | テンプレート、画像、ひな形などを置く |
| `references/` | 詳細な補足資料や長い参照ドキュメントを置く |

このプロジェクトの `.agents/skills` 配下では、`SKILL.md` と `agents/openai.yaml` を置く構成にしています。

すべてのスキルに `scripts/` や `assets/` が必要なわけではありません。

まずは `SKILL.md` だけで始め、必要になったら補助ファイルを足していく考え方で十分です。

## scriptsを使う意味

スキルはMarkdownの手順書としても使えますが、固定の操作は `scripts/` に切り出すと便利です。

たとえば、毎回同じ検証コマンドを実行する、同じ形式でファイルを生成する、長いチェック処理を行う、といった作業はスクリプトにできます。

```txt
毎回AIが長い手順を考えて実行する
  ↓
固定処理をscriptsにする
  ↓
AIは必要なスクリプトを呼び出すだけでよくなる
```

スクリプト化すると、次のようなメリットがあります。

| メリット | 内容 |
| --- | --- |
| 手順の安定 | 毎回同じ処理を実行しやすい |
| ミスの削減 | 長い手作業やコマンドの打ち間違いを減らせる |
| トークン節約 | AIが毎回長い処理内容を生成しなくて済む |
| 保守しやすい | 処理を直すときにスクリプトだけを更新できる |

特に、固定の変換処理、検証処理、ファイル生成処理は、文章で毎回説明するよりもスクリプトとして置いた方が安定します。

> スキルは「手順を文章で教える」だけでなく、「固定処理をスクリプトとして渡す」ことで、AIの作業をより安定させられます。

## skill-creatorを使うと作りやすい

Agent skillsはMarkdownで書かれています。

そのため、最終的には `SKILL.md` のようなファイルとして管理されます。

ただし、最初から自分でMarkdownをきれいに書き切る必要はありません。

`skill-creator` を使うと、何をスキル化したいかを対話しながら整理し、スキルの形にまとめやすくなります。

```txt
スキル化したい作業を相談する
  ↓
使う場面や手順を整理する
  ↓
skill-creatorでSKILL.mdの形にする
```

つまり、「スキルを作るには難しいMarkdownを全部自分で書かないといけない」と考えなくて大丈夫です。

よく繰り返す作業や、毎回説明している判断基準が見えてきたら、それを `skill-creator` でスキル化する候補にできます。

## どんな作業をスキル化するとよいか

スキル化に向いているのは、繰り返し使う作業です。

| スキル化に向いているもの | 理由 |
| --- | --- |
| 毎回同じ手順で進める作業 | 手順漏れを減らしやすい |
| 判断基準がある作業 | レビュー品質をそろえやすい |
| プロジェクト固有の作法 | AIに前提を渡しやすい |
| 検証コマンドが決まっている作業 | 完了条件を明確にしやすい |

逆に、1回限りの雑談や、その場で考えればよい軽い相談は、無理にスキル化しなくてもよいです。

## まず押さえること

- Agent skillsは、AIに特定作業の進め方を渡すための手順書
- 毎回同じ説明をしなくても、作業手順や判断基準を再利用しやすくなる
- スキルには、使う場面、手順、判断基準、参照ファイル、完了前の確認を書く
- 社内特有の暗黙知や判断基準を、AIが参照できる形に言語化できる
- スキルは `SKILL.md` を中心に、`agents/openai.yaml` や必要に応じて `scripts/`、`assets/` を持てる
- 固定処理を `scripts/` にすると、手順が安定し、トークン消費も抑えやすい
- `skill-creator` を使えば、対話しながらスキルを作りやすい
- 繰り返し使う作業やプロジェクト固有の作法はスキル化に向いている

> Agent skillsは、AIとの共同作業を「その場限りの会話」から「再利用できる作業手順」へ近づける仕組みです。

## 理解度チェック

Q1. Agent skillsの説明として最も適切なものはどれですか。

- A. AIに特定作業の手順や判断基準を渡すための再利用できる手順書
- B. Gitのコミット履歴を自動で削除する仕組み
- C. 画像生成AIのノイズを取り除くためだけの設定
- D. APIキーを公開してツール連携を簡単にする仕組み

解説: Agent skillsは、AIに特定作業の進め方や判断基準を渡すための仕組みです。

Q2. スキル化に向いている作業として最も適切なものはどれですか。

- A. 1回だけの雑談で、今後繰り返す予定がないもの
- B. 毎回手順や判断基準を説明している繰り返し作業
- C. AIに一切前提を渡したくない作業
- D. 検証や完了条件がまったく存在しない作業

解説: 繰り返し使う手順や判断基準は、スキル化すると再利用しやすくなります。

Q3. skill-creatorの使い方として最も近いものはどれですか。

- A. すべてのMarkdownを人間が手書きしないと使えないようにする
- B. 既存スキルを必ず削除するためだけに使う
- C. スキル化したい作業を対話で整理し、`SKILL.md` の形にまとめやすくする
- D. 学習コンテンツのチェックテストを自動で全削除する

解説: `skill-creator` を使うと、スキル化したい作業の使いどころや手順を整理し、スキルとして作りやすくなります。

Q4. スキルに書く内容として適切なものはどれですか。

- A. AIに読ませたくない秘密情報やAPIキー
- B. 何の作業に使うか分からない曖昧な雑談だけ
- C. 検証コマンドを使う必要がある作業でも、完了条件は書かない
- D. 使う場面、作業手順、判断基準、完了前の確認

解説: スキルには、いつ使うか、どう進めるか、何を確認するかを書くと使いやすくなります。

Q5. このプロジェクトで使っているスキルの基本的な構成として最も適切なものはどれですか。

- A. `SKILL.md` を中心にし、`agents/openai.yaml` や必要に応じて `scripts/`、`assets/` を持つ
- B. 必ず画像ファイルだけで構成し、Markdownは使わない
- C. すべての処理を毎回プロンプトだけで説明し、ファイルは作らない
- D. APIキーやパスワードを直接書くためのディレクトリである

解説: このプロジェクトでは `SKILL.md` と `agents/openai.yaml` を置き、必要に応じて `scripts/`、`assets/`、`references/` などを追加します。

Q6. `scripts/` を使うメリットとして最も適切なものはどれですか。

- A. スクリプトを置くと、AIの出力確認が完全に不要になる
- B. `scripts/` に処理を置くと、固定処理を再利用できなくなる
- C. `scripts/` は秘密情報を公開するためだけに使う
- D. 固定処理をスクリプト化することで、手順を安定させ、AIが毎回長い処理を生成する負担を減らせる

解説: 固定の検証、変換、生成処理をスクリプトにすると、手順の安定化やトークン消費の削減につながります。

Q7. 社内ナレッジをスキル化する意味として最も適切なものはどれですか。

- A. AIの出力確認が完全に不要になる
- B. 社内特有の作法や判断基準を、AIが参照できる形に言語化しやすくなる
- C. すべての作業を必ず同じ答えに固定できる
- D. 権限管理や検証を考えなくてよくなる

解説: Agent skillsを使うと、社内の暗黙知や判断基準をAIが参照できる形にし、複数の作業場面で再利用しやすくなります。

答え:

- Q1: A
- Q2: B
- Q3: C
- Q4: D
- Q5: A
- Q6: D
- Q7: B
