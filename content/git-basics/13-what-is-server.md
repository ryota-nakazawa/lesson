# サーバーって何？

サーバーは、他のコンピューターやアプリからのリクエストに応じて、情報や機能を提供する役割を持つものです。

たとえば、ブラウザでWebサイトを開くとき、私たちのPCやスマートフォンは「このページを見せてください」とリクエストします。そのリクエストを受け取り、HTMLや画像、データなどを返す側がサーバーです。

```txt
ブラウザ
  ↓ ページをください
サーバー
  ↓ HTMLやデータを返す
ブラウザに画面が表示される
```

> まとめ: サーバーは、リクエストを受け取り、必要な情報や機能を返す役割です。

## サーバーは物理的な機械だけではない

「サーバー」と聞くと、大きな機械が並んでいる場所を想像するかもしれません。

それも間違いではありません。実際に、データセンターには多くのサーバー用コンピューターがあります。

ただし、開発で「サーバー」と言うときは、物理的な機械だけを指すとは限りません。

次のような意味で使われます。

- サーバー用のコンピューター
- クラウド上で動いている実行環境
- リクエストを受けてレスポンスを返すソフトウェア
- 自分のPC上で一時的に動かす開発用サーバー

つまり、サーバーは「機械の名前」というより、**リクエストに対してレスポンスを返す役割** と考えると理解しやすくなります。

## クライアントとサーバー

サーバーを理解するときは、クライアントとセットで考えます。

クライアントは、サーバーにリクエストする側です。

サーバーは、リクエストを受けてレスポンスする側です。

```txt
クライアント: リクエストする側
サーバー: レスポンスを返す側
```

Webサービスでは、ブラウザやスマートフォンアプリがクライアントになることが多いです。

```txt
ブラウザ / アプリ
  ↓ リクエスト
サーバー
  ↓ レスポンス
ブラウザ / アプリ
```

たとえば、ログイン画面でメールアドレスとパスワードを送信すると、クライアントがサーバーに情報を送ります。サーバーは内容を確認し、ログインできるかどうかを返します。

## Webサーバーとアプリケーションサーバー

サーバーにはいろいろな種類があります。

初学者のうちは、まず次の2つを知っておくと十分です。

```txt
Webサーバー: ページやファイルを返す
アプリケーションサーバー: 処理や判定を行う
```

Webサーバーは、HTML、CSS、画像などを返す役割を持ちます。

アプリケーションサーバーは、ログイン判定、データ登録、検索、計算など、サービスの中身の処理を担当します。

実際のサービスでは、これらの役割が分かれていることもあれば、1つの仕組みの中でまとめて動いていることもあります。

ここでは、細かい分類を完璧に覚えるよりも、次のように理解できれば十分です。

- 画面やファイルを返す役割がある
- リクエストに応じて処理する役割がある
- サーバー側で動く処理がある

## ローカルサーバー

ローカルサーバーは、自分のPC上で動かすサーバーです。

開発中は、作ったWebアプリをすぐ確認するために、自分のPCでサーバーを起動することがあります。

たとえば、次のようなコマンドを実行することがあります。

```bash
npm run dev
```

すると、ターミナルに次のような表示が出ることがあります。

```txt
Local: http://localhost:5173/
```

この `localhost` は、自分のPCを指す特別な名前です。

つまり、`http://localhost:5173/` にアクセスすると、インターネット上のどこかのサーバーではなく、自分のPC上で動いている開発用サーバーにアクセスしています。

```txt
自分のブラウザ
  ↓ http://localhost:5173/
自分のPC上のローカルサーバー
```

ローカルサーバーは、開発や確認のために使います。基本的には、自分のPC上で確認するためのものです。

## リモートサーバー

リモートサーバーは、自分のPCではなく、別の場所で動いているサーバーです。

たとえば、クラウド上のサーバーや、会社が管理しているサーバーがリモートサーバーです。

Webサービスを利用者に公開する場合、アプリケーションはリモートサーバー上で動いていることが多いです。

```txt
利用者のブラウザ
  ↓ https://example.com
リモートサーバー
  ↓ ページやデータを返す
利用者のブラウザ
```

リモートサーバーは、チームメンバーや利用者など、自分以外の人もアクセスする前提で使われます。

## ローカルサーバーとリモートサーバーの違い

ローカルサーバーとリモートサーバーは、どちらもサーバーですが、使う目的が違います。

```txt
ローカルサーバー: 自分のPCで開発・確認するため
リモートサーバー: チームや利用者がアクセスするため
```

たとえば、開発中の画面を自分だけで確認するならローカルサーバーで十分です。

一方で、チームに共有したり、利用者に公開したりするには、リモートサーバーやクラウド上の環境に反映する必要があります。

この「作ったものを利用者がアクセスできる場所に反映すること」が、後のレッスンで扱うデプロイにつながります。

> ローカルサーバーで動くことと、リモートサーバーで正しく動くことは別です。本番環境では設定、データ、ネットワーク条件が違うことがあります。

## 開発で「サーバーを立てる」とは

開発現場では、「サーバーを立てる」という表現を使うことがあります。

これは、多くの場合、サーバーとして動くソフトウェアを起動し、リクエストを受け取れる状態にする、という意味です。

たとえば、ローカルでWebアプリを確認するために、次のような流れになります。

```txt
コードを書く
  ↓
開発用サーバーを起動する
  ↓
ブラウザでlocalhostにアクセスする
  ↓
画面を確認する
```

このときのサーバーは、必ずしも大きな機械を用意するという意味ではありません。自分のPC上で、サーバーとして動くプログラムを起動しているだけの場合もあります。

## まず押さえること

サーバーについて、最初に押さえるべきことは次の4つです。

- サーバーはリクエストに応じて情報や機能を返す役割
- クライアントはリクエストする側、サーバーはレスポンスする側
- ローカルサーバーは自分のPCで開発確認するために使う
- リモートサーバーはチームや利用者がアクセスする場所で動く

> サーバーを理解すると、API、デプロイ、インフラ、本番環境の話がつながりやすくなります。

## 理解度チェック

Q1. サーバーの説明として最も近いものはどれですか。

- A. 画面上のボタンだけを作る道具
- B. Gitのコミットメッセージを保存する場所
- C. リクエストを受け取り、必要な情報や機能を返す役割
- D. ブラウザの表示色を変える設定

解説: サーバーは、クライアントからのリクエストに応じて、HTMLやデータなどを返す役割です。

Q2. クライアントとサーバーの関係として最も近いものはどれですか。

- A. クライアントはリクエストする側、サーバーはレスポンスを返す側
- B. クライアントは常にデータベース、サーバーは常に画像ファイル
- C. クライアントはGitHub、サーバーはコミットメッセージ
- D. クライアントとサーバーは同じ意味で使われる

解説: Webサービスでは、ブラウザやアプリがリクエストし、サーバーがレスポンスを返す関係になります。

Q3. ローカルサーバーの説明として最も近いものはどれですか。

- A. 利用者全員がアクセスする本番用のサーバー
- B. 必ず会社のデータセンターに置かれている機械
- C. Gitのリモートリポジトリ
- D. 自分のPC上で開発や確認のために動かすサーバー

解説: ローカルサーバーは、自分のPC上で開発中のWebアプリなどを確認するために使います。

Q4. 開発で「サーバーを立てる」という表現の意味として最も近いものはどれですか。

- A. 物理的なサーバーラックを必ず購入すること
- B. サーバーとして動くソフトウェアを起動し、リクエストを受け取れる状態にすること
- C. ブラウザのタブを閉じること
- D. すべてのファイルを削除すること

解説: 開発では、自分のPC上でサーバーとして動くプログラムを起動する場合にも「サーバーを立てる」と言うことがあります。

答え:

- Q1: C
- Q2: A
- Q3: D
- Q4: B
