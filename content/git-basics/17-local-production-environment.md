# ローカル環境と本番環境

アプリケーション開発では、同じアプリでも **どこで動かすか** によって環境を分けて考えます。

たとえば、自分のPCで確認する環境と、実際の利用者が使う環境は分けて扱います。

| 環境 | 何のために使うか |
| --- | --- |
| ローカル環境 | 自分のPCで開発・確認する |
| sandbox（サンドボックス）環境 | 外部APIなどを安全に試す |
| ステージング環境 | 本番に近い状態で最終確認する |
| 本番環境 | 実際の利用者が使う |

> まとめ: ローカルで動くことと、本番で安全に動くことは別です。

## ローカル環境

ローカル環境では、開発中のアプリを自分のPCで動かします。

```bash
npm run dev
```

このようなコマンドでローカルサーバーを立て、`localhost` で確認することがあります。

ローカル環境では、まだ完成していない変更を自由に試せます。画面を崩しても、エラーを出しても、基本的には自分のPCの中だけで影響が止まります。

```txt
自分のPC
  ↓
localhost で確認
  ↓
問題なければ次の環境へ
```

ただし、ローカル環境で動いたからといって、本番環境でも必ず同じように動くとは限りません。OS、Node.jsのバージョン、環境変数、外部APIの接続先などが違うことがあるためです。

## sandbox（サンドボックス）環境

sandbox（サンドボックス）環境は、本番データや本番サービスに影響を与えずに試すための環境です。

特に、外部APIや決済サービス、メール配信サービスなどでは、sandbox環境と本番環境が分かれていることがよくあります。

| 環境 | 用途 | 例 |
| --- | --- | --- |
| sandbox環境 | テスト用に安全に試す | テスト用APIキー、テスト用データ、実際には送信されない決済 |
| 本番環境 | 実際の利用者向けに動かす | 本番APIキー、本番データ、実際に処理される決済 |

たとえば、決済APIを使うアプリで、開発中に本番の決済APIへ接続してしまうと、テストのつもりで実際の請求が発生する可能性があります。

そのため、開発中はsandbox用のAPIに接続し、本番公開時だけ本番用のAPIに接続するように分けます。

```txt
開発中
  ↓
sandbox API に接続
  ↓
テスト用の処理だけが行われる

本番公開後
  ↓
prod API に接続
  ↓
実際の利用者向けの処理が行われる
```

> sandboxは「本番に影響を出さずに試す場所」と考えると分かりやすいです。

## 本番環境

本番環境は、利用者が実際に使う環境です。

本番環境では、次のような点が重要になります。

- 利用者に影響する
- データを安全に扱う必要がある
- エラーや停止を避ける必要がある
- 設定や権限を慎重に扱う必要がある
- 外部APIや決済などが実際に動く

本番環境は、英語で **production environment** と呼ばれます。短く **prod** と表現されることもあります。

たとえば、`prod API`、`prod DB`、`production server` のように書かれていたら、本番向けの接続先や環境を指していることが多いです。

## 環境ごとに接続先が変わる

環境を分けるときに重要なのが、接続先や設定も環境ごとに分けることです。

たとえば、APIの接続先は次のように分かれることがあります。

| 環境 | APIの接続先の例 | 使うデータ |
| --- | --- | --- |
| ローカル環境 | `http://localhost:3000` | 自分のPC上のデータ |
| sandbox環境 | `https://sandbox-api.example.com` | テスト用データ |
| 本番環境 | `https://api.example.com` | 実際の利用者データ |

同じアプリでも、環境によって参照するAPIやデータベースが違います。

この切り替えには、環境変数が使われることがあります。

```txt
API_URL=https://sandbox-api.example.com
```

本番では、同じ `API_URL` という名前でも値が変わります。

```txt
API_URL=https://api.example.com
```

アプリのコードは同じでも、環境変数の値を変えることで、sandbox向けに動かしたり、本番向けに動かしたりできます。

> 環境変数は、環境ごとに変わる設定をアプリの外から渡すためによく使われます。

## 環境が分かれる理由

開発中の変更をいきなり本番に出すと、利用者に不具合が見えてしまうことがあります。また、テストのつもりで本番APIや本番データを触ってしまうと、実際の利用者に影響が出る可能性があります。

そのため、ローカル環境、プレビュー環境、ステージング環境、本番環境のように分けて確認することがあります。

```txt
ローカル
  ↓
sandbox / プレビュー
  ↓
ステージング
  ↓
本番
```

```txt
自分のPCで開発
  ↓
sandbox / プレビューで安全に確認
  ↓
ステージングで本番に近い状態を確認
  ↓
prod環境で利用者に提供
```

> 環境を分けることで、確認中の変更と利用者向けの安定した状態を分けて扱えます。

## FDE人材が押さえること

FDE人材としては、細かいインフラ設定をすべて書ける必要はなくても、どの環境の話をしているのかを区別できることが重要です。

特に、生成AIや開発者に相談するときは、次のように伝えられると状況が整理しやすくなります。

| 伝え方 | 意味 |
| --- | --- |
| ローカルでは動くが、本番ではエラーになる | 自分のPCと本番環境で差分があるかもしれない |
| sandbox APIでは成功するが、prod APIでは失敗する | APIキー、権限、データ、接続先の違いが関係しているかもしれない |
| stagingでは確認済みで、本番反映前です | 本番に近い環境で確認している段階 |

環境を区別できると、「どこで起きている問題か」「どの接続先を見ればよいか」「本番に影響があるか」を判断しやすくなります。

## まず押さえること

最初に押さえるべきことは次の4つです。

- ローカル環境は、自分のPCで開発・確認する場所
- sandbox環境は、本番に影響を出さずに外部APIなどを試す場所
- 本番環境、つまりprod環境は、実際の利用者が使う場所
- 環境ごとにAPI、DB、APIキー、環境変数が分かれることが多い

> 環境の違いが分かると、「どこで動いているのか」「どのデータに接続しているのか」を切り分けやすくなります。

## 理解度チェック

Q1. ローカル環境の説明として最も近いものはどれですか。

- A. 実際の利用者がアクセスする公開済みの環境
- B. 外部APIの本番データだけを操作する環境
- C. 自分のPC上で開発中のアプリを確認する環境
- D. 本番と同じAPIキーだけを管理する環境

解説: ローカル環境は、自分のPCで開発中のアプリを動かし、`localhost` などで確認する環境です。

Q2. sandbox環境を使う理由として最も近いものはどれですか。

- A. 本番データや本番サービスに影響を与えずに外部APIなどを試すため
- B. 利用者にそのまま公開するため
- C. ローカル環境での確認をすべて禁止するため
- D. 環境変数を使わずにすべての設定を固定するため

解説: sandbox環境は、テスト用APIキーやテスト用データを使い、本番に影響を出さずに試すための環境です。

Q3. `API_URL` を環境ごとに変える目的として最も近いものはどれですか。

- A. アプリの画面デザインを自動で変更するため
- B. 同じコードでも、sandbox用APIや本番用APIなど接続先を切り替えるため
- C. Gitのブランチ名を自動で変更するため
- D. Node.jsのバージョンを必ず最新にするため

解説: 環境変数を使うと、同じアプリのコードでも環境ごとにAPI接続先などの設定を切り替えられます。

Q4. 「sandbox APIでは成功するが、prod APIでは失敗する」ときに考えるべきこととして最も近いものはどれですか。

- A. sandboxとprodは常に完全に同じなので、確認することはない
- B. ローカル環境の画面色だけが原因である
- C. 本番環境では外部APIが一切使われない
- D. prod API側のAPIキー、権限、データ、接続先がsandboxと違う可能性がある

解説: sandboxとprodでは、APIキー、権限、データ、接続先が異なることがあるため、どの環境で失敗しているかを切り分けることが重要です。

答え:

- Q1: C
- Q2: A
- Q3: B
- Q4: D
