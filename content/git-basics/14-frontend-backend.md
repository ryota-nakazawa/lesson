# フロントエンド / バックエンド

Webアプリケーションは、大きく **フロントエンド** と **バックエンド** に分けて考えることができます。

フロントエンドは、利用者が直接見る画面や操作部分です。

バックエンドは、画面の裏側でデータを処理したり、保存したり、必要な情報を返したりする部分です。

```txt
利用者
  ↓ 操作する
フロントエンド
  ↓ APIでリクエスト
バックエンド
  ↓ データ取得・保存
データベース
```

> まとめ: フロントエンドは利用者に近い部分、バックエンドは裏側で処理やデータを扱う部分です。

## フロントエンドとは

フロントエンドは、利用者が直接触れる部分です。

たとえば、次のようなものがフロントエンドに含まれます。

- 画面のレイアウト
- ボタンや入力欄
- メニューやナビゲーション
- エラーメッセージ
- 読み込み中の表示
- スマートフォンやPCでの見え方

WebサイトやWebアプリを開いたときに、ブラウザ上で見えている部分の多くがフロントエンドです。

フロントエンドでは、HTML、CSS、JavaScript、Reactなどの技術が使われることがあります。

ただし、初学者のうちは技術名を覚えるよりも、まず「利用者が見る・触る部分」と考えると分かりやすいです。

## UIとUXの観点

フロントエンドでは、見た目だけでなく **UI** と **UX** の観点も重要です。

UIは、User Interfaceの略です。利用者が触れる画面や部品のことです。

たとえば、ボタン、入力欄、チェックボックス、メニュー、通知表示などがUIにあたります。

UXは、User Experienceの略です。利用者がそのサービスを使うときの体験全体のことです。

たとえば、次のような観点がUXに関わります。

- 迷わず操作できるか
- 入力ミスに気づきやすいか
- エラーが起きたときに次の行動が分かるか
- 表示が遅すぎないか
- スマートフォンでも使いやすいか

フロントエンドは、単にきれいな画面を作るだけではありません。利用者が目的を達成しやすいように、情報の見せ方や操作の流れを設計する役割があります。

```txt
UI: 画面上の部品や見た目
UX: 使っているときの分かりやすさ・快適さ
```

> フロントエンドでは、「見た目が整っているか」だけでなく、「利用者が迷わず使えるか」も大切です。

## バックエンドとは

バックエンドは、画面の裏側で動く処理部分です。

たとえば、次のようなものがバックエンドに含まれます。

- ログインできるか判定する
- フォームで送信された内容を保存する
- データベースから情報を取得する
- APIとしてデータを返す
- 権限を確認する
- 決済や通知などの処理を行う

利用者からは直接見えにくいですが、サービスを正しく動かすために重要な部分です。

たとえば、ログイン画面でメールアドレスとパスワードを入力したとします。

```txt
フロントエンド: 入力欄とログインボタンを表示する
バックエンド: 入力内容を確認し、ログインできるか判定する
```

画面はフロントエンドが担当し、ログイン判定やデータ確認はバックエンドが担当します。

## APIでつながる

フロントエンドとバックエンドは、APIを通じてやりとりすることが多いです。

フロントエンドは、必要な情報をバックエンドにリクエストします。

バックエンドは、処理した結果やデータをレスポンスとして返します。

```txt
フロントエンド
  ↓ ユーザー一覧をください
バックエンド API
  ↓ ユーザー一覧データを返す
フロントエンド
  ↓ 画面に表示する
```

たとえば、一覧画面を開いたとき、フロントエンドがバックエンドに「一覧データをください」とリクエストします。バックエンドはデータベースから情報を取り出し、フロントエンドへ返します。

フロントエンドは、返ってきたデータを使って画面を表示します。

## データベースとの関係

バックエンドは、データベースとやりとりすることが多いです。

データベースは、ユーザー情報、注文情報、記事、設定などを保存する場所です。

```txt
フロントエンド
  ↓ APIリクエスト
バックエンド
  ↓ データを取得・保存
データベース
```

フロントエンドが直接データベースを操作するのではなく、バックエンドを通してデータを扱う構成がよく使われます。

これは、認証、権限、入力チェック、セキュリティなどをバックエンド側で管理するためです。

## 役割は分かれていても連携している

フロントエンドとバックエンドは役割が違いますが、別々に存在しているだけではサービスになりません。

たとえば、問い合わせフォームを考えます。

```txt
フロントエンド:
入力画面を表示する
入力内容をチェックする
送信ボタンを押せるようにする

バックエンド:
送信内容を受け取る
必要なチェックをする
データベースに保存する
通知を送る
```

フロントエンドだけでは、入力画面は作れても、送信内容を安全に保存できません。

バックエンドだけでは、利用者が操作する画面がありません。

そのため、Webアプリケーションでは、フロントエンドとバックエンドが連携して1つの体験を作ります。

## ディレクトリの切り方の例

実際に実装するときは、フロントエンドとバックエンドをディレクトリで分けることがあります。

たとえば、次のような構成です。

```txt
team-app/
  frontend/
    src/
    package.json
  backend/
    src/
    package.json
  README.md
```

この例では、`frontend/` が画面側、`backend/` がサーバー側です。

```txt
frontend/: 利用者が見る画面やUI
backend/: API、認証、保存処理などの裏側
```

フロントエンド側には、画面部品、ページ、スタイル、APIを呼び出す処理などを置きます。

```txt
frontend/
  src/
    pages/
    components/
    styles/
    api/
```

バックエンド側には、APIの入口、処理のロジック、データベースとやりとりする処理などを置きます。

```txt
backend/
  src/
    routes/
    services/
    database/
```

このように分けておくと、「画面の問題なのか」「APIや保存処理の問題なのか」を切り分けやすくなります。

別の構成として、1つのプロジェクト内で `app/` や `api/` のように分けることもあります。

```txt
app/
  pages/
  components/
api/
  users/
  login/
```

ディレクトリ名や切り方は、使うフレームワークやチームのルールによって変わります。大切なのは、どこが画面側で、どこが裏側の処理なのかを読み取れることです。

> 生成AIに依頼するときも、「`frontend/` のログイン画面を直して」「`backend/` のログインAPIを確認して」のように伝えられると、対象を絞りやすくなります。

## FDE人材が押さえるべきこと

FDE人材としては、細かい実装をすべて書ける必要がなくても、役割の違いを理解しておくことが重要です。

特に、生成AIに開発を依頼するときは、どちらの話をしているのかを分けて伝えると精度が上がります。

たとえば、次のように分けて考えます。

```txt
画面の文言やボタン配置を変えたい
→ フロントエンドの話

ログイン判定や保存処理を変えたい
→ バックエンドの話

表示するデータの中身を変えたい
→ APIやデータベースの話も関係する
```

問題が起きたときも、どこで起きているのかを切り分けやすくなります。

- 画面が崩れているのか
- ボタンを押してもAPIが呼ばれていないのか
- APIは呼ばれているがエラーになっているのか
- データベースに保存されていないのか

この切り分けができると、開発者や生成AIに相談するときの説明が具体的になります。

## まず押さえること

最初に押さえるべきことは次の4つです。

- フロントエンドは利用者が見る・触る部分
- フロントエンドではUI/UXの観点が重要
- バックエンドは裏側で処理やデータを扱う部分
- フロントエンドとバックエンドはAPIを通じて連携することが多い

> フロントエンドとバックエンドの違いが分かると、Webアプリのどこを直したいのか、どこで問題が起きているのかを説明しやすくなります。

## 理解度チェック

Q1. フロントエンドの説明として最も近いものはどれですか。

- A. 利用者が見る画面や操作部分
- B. データベースだけを管理する場所
- C. サーバーの物理的な機械だけを指す言葉
- D. Gitのブランチ名

解説: フロントエンドは、画面、ボタン、入力欄など、利用者が直接見る・触る部分です。

Q2. UXの説明として最も近いものはどれですか。

- A. サーバーのIPアドレス
- B. Gitのコミット履歴
- C. 利用者がサービスを使うときの体験全体
- D. データベースの表名

解説: UXはUser Experienceの略で、利用者が迷わず快適に目的を達成できるかなどに関わります。

Q3. バックエンドの役割として最も近いものはどれですか。

- A. ボタンの色だけを調整する
- B. ログイン判定、データ保存、APIとしてのデータ返却などを行う
- C. ブラウザのタブを並べ替える
- D. 画像の明るさだけを変える

解説: バックエンドは、画面の裏側で処理、認証、データ取得・保存などを担当します。

Q4. `frontend/` と `backend/` に分けるメリットとして最も近いものはどれですか。

- A. すべての処理を必ず画面側だけで完結できる
- B. Gitを使わなくてよくなる
- C. APIが不要になる
- D. 画面側の問題か、裏側の処理の問題かを切り分けやすくなる

解説: ディレクトリを分けると、画面側と裏側の処理の責務が見えやすくなり、問題の切り分けもしやすくなります。

答え:

- Q1: A
- Q2: C
- Q3: B
- Q4: D
