# MCPって何？

MCPは、Model Context Protocolの略です。

AIエージェントが外部のツール、データ、サービスに接続するための共通ルールとして使われます。

> まとめ: MCPは、AIが外部ツールを使うための接続ルールです。

## なぜMCPが必要か

AIにできることは、会話だけではありません。

外部ツールとつながることで、次のようなことができます。

| つながる先 | できることの例 |
| --- | --- |
| ローカルファイル | ファイルを読む、編集する |
| GitHub | IssueやPull Requestを確認する |
| Slack | メッセージを検索する、投稿する |
| Google Drive | ドキュメントやスプレッドシートを読む |
| DBや社内システム | データを検索する、更新する |

ただし、ツールごとに接続方法がバラバラだと、AIエージェント側の実装が複雑になります。

MCPがない場合、GitHub、Slack、Google Drive、社内DBなど、それぞれのAPI仕様に合わせて個別に接続処理を書く必要があります。

| 接続先 | 個別に合わせる必要があるもの |
| --- | --- |
| GitHub API | 認証方法、IssueやPRの取得方法、レスポンス形式 |
| Slack API | チャンネル検索、メッセージ投稿、権限スコープ |
| Google Drive API | ファイル検索、DocsやSheetsの扱い、共有権限 |
| 社内API | 独自の認証、独自のデータ形式、独自のエラー処理 |

この状態だと、AIエージェントを作る側は、ツールを増やすたびにAPI仕様を読み、それぞれの呼び出し方やエラー処理を実装しなければなりません。

MCPは、AIと外部ツールの接続方法をそろえるための仕組みです。

```txt
AIエージェント
  ↓ MCP
外部ツール / データ / サービス
```

## MCPサーバーとクライアント

MCPでは、AIエージェント側とツール提供側を分けて考えます。

| 役割 | 意味 |
| --- | --- |
| MCPクライアント | MCPサーバーを呼び出す側。AIエージェントやAIアプリ側 |
| MCPサーバー | ツールやデータを提供する側 |

たとえば、AIエージェントがGoogle Driveの資料を読む場合、AIエージェント側がMCPクライアント、Google Driveに接続する仕組みを持つ側がMCPサーバーです。

```txt
MCPクライアント
  ↓
MCPサーバー
  ↓
Google Drive / GitHub / DB など
```

## APIがあるからMCPを実現できる

MCPは、外部サービスそのものを作り直す仕組みではありません。

多くの場合、MCPサーバーの裏側では、既存のAPIを呼び出しています。

たとえば、GitHubにはIssueやPull Requestを取得するAPIがあります。Slackにもメッセージ検索や投稿のAPIがあります。

MCPサーバーは、それらのAPIをAIエージェントが使いやすい形で包みます。

```txt
AIエージェント
  ↓ MCP
MCPサーバー
  ↓ API
GitHub / Slack / Google Drive など
```

つまり、APIがあることで、MCPサーバーは外部サービスとやりとりできます。

MCPは「APIの代わり」ではなく、AIエージェントがAPIやツールを扱いやすくするための共通の入り口と考えると分かりやすいです。

| MCPがない場合 | MCPがある場合 |
| --- | --- |
| ツールごとにAPI仕様を直接理解して接続する | MCPの共通ルールに沿ってツールを呼び出せる |
| 認証、入力形式、出力形式、エラー処理がバラバラ | MCPサーバー側で違いを吸収しやすい |
| 接続先が増えるほどAIエージェント側が複雑になる | AIエージェント側の接続方法をそろえやすい |

## ローカルMCP

ローカルMCPは、自分のPC上で動くMCPサーバーです。

たとえば、AIエージェントがローカルファイル、ターミナル、開発中のプロジェクトにアクセスする場合に使われます。

| 例 | できること |
| --- | --- |
| ファイル操作MCP | ローカルファイルを読む、編集する |
| Git操作MCP | ローカルリポジトリの状態を確認する |
| 開発ツールMCP | テスト実行やビルド結果を確認する |

ローカルMCPは、自分のPC上の情報にアクセスできるため便利です。

一方で、ファイル編集やコマンド実行など、強い権限を持つことがあります。どのディレクトリにアクセスできるか、どの操作を許可するかを絞ることが重要です。

## リモートMCP

リモートMCPは、インターネット上や社内サーバー上で動くMCPサーバーです。

AIエージェントは、ネットワーク経由でそのMCPサーバーに接続します。

| 例 | できること |
| --- | --- |
| SaaS連携MCP | Slack、GitHub、Notionなどと連携する |
| 社内システムMCP | 社内DBや業務システムを検索する |
| 共通ツールMCP | チーム全員で同じ接続先を使う |

リモートMCPは、チームで共通のツール接続を使いやすい点が便利です。

一方で、認証、権限管理、通信の安全性が重要になります。誰がどのデータにアクセスできるかを設計しないと、情報漏えいにつながる可能性があります。

## ローカルMCPとリモートMCPの違い

ローカルMCPとリモートMCPは、どこでMCPサーバーが動くかが違います。

| 観点 | ローカルMCP | リモートMCP |
| --- | --- | --- |
| 動く場所 | 自分のPC | サーバーやクラウド |
| 主な用途 | ローカルファイル、開発環境、ターミナル連携 | SaaS、社内システム、チーム共通ツール |
| メリット | 手元の環境に直接アクセスしやすい | チームで共通利用しやすい |
| 注意点 | PC内のファイルやコマンド権限に注意 | 認証、権限、通信の安全性に注意 |

どちらが優れているというより、使いたいデータやツールがどこにあるかで使い分けます。

## まず押さえること

- MCPは、AIエージェントが外部ツールやデータに接続するための共通ルール
- MCPクライアントはAIエージェント側、MCPサーバーはツールやデータを提供する側
- MCPサーバーの裏側では、既存のAPIを呼び出していることが多い
- MCPがないと、ツールごとのAPI仕様に合わせて個別に接続する必要がある
- ローカルMCPは自分のPC上で動き、ローカルファイルや開発環境と相性がよい
- リモートMCPはサーバー上で動き、SaaSや社内システムとの連携に向いている
- MCPを使うときは、認証、権限、アクセス範囲を設計する

> MCPは、AIエージェントが外部世界とやりとりするための橋渡しです。便利な反面、どこまでアクセスを許可するかが重要になります。

## 理解度チェック

Q1. MCPの説明として最も適切なものはどれですか。

- A. AIエージェントが外部ツールやデータに接続するための共通ルール
- B. Gitのコミット履歴を削除するコマンド
- C. 画像生成AIのノイズ除去アルゴリズム
- D. DNSでドメイン名をIPアドレスに変換する仕組み

解説: MCPは、AIエージェントが外部ツールやデータに接続するための共通ルールです。

Q2. APIとMCPの関係として最も適切なものはどれですか。

- A. MCPがあるとAPIは一切不要になる
- B. MCPサーバーの裏側で既存APIを呼び出し、APIごとの差分をAIが扱いやすい形に包むことが多い
- C. APIは画像ファイル専用で、MCPとは関係しない
- D. MCPはAPIを禁止するための仕組みである

解説: MCPサーバーは、GitHubやSlackなどの既存APIを呼び出し、認証や入力・出力形式の違いをAIエージェントが扱いやすい形にすることがあります。

Q3. ローカルMCPの説明として最も適切なものはどれですか。

- A. 必ずクラウド上だけで動き、手元のPCにはアクセスしないMCP
- B. 自分のPC上で動き、ローカルファイルや開発環境にアクセスできるMCP
- C. 認証や権限管理が一切不要なMCP
- D. SaaSのデータだけを読むためのMCP

解説: ローカルMCPは自分のPC上で動き、ローカルファイルや開発環境と連携しやすい一方、アクセス範囲に注意が必要です。

Q4. リモートMCPを使うときの注意点として最も適切なものはどれですか。

- A. サーバー上で動くため、認証や権限設計を考える必要がある
- B. インターネット経由で動くので、誰でも全データにアクセスできるようにする
- C. リモートMCPではAPIキーや認証情報を公開しておく
- D. チーム共通利用には向かないため、必ず個人PCだけで動かす

解説: リモートMCPはサーバーやクラウド上で動くため、認証、権限、通信の安全性を設計する必要があります。

Q5. MCPを使うときの考え方として最も適切なものはどれですか。

- A. 便利なので、AIにすべてのファイルと全APIの権限を渡す
- B. MCPを使えば、AIの出力確認は不要になる
- C. MCPは外部ツールと接続しないため、権限を考えなくてよい
- D. 使いたいデータやツールに応じて、ローカルMCPとリモートMCPを使い分ける

解説: MCPは外部ツールと接続するための仕組みなので、用途に応じた使い分けと権限設計が重要です。

答え:

- Q1: A
- Q2: B
- Q3: B
- Q4: A
- Q5: D
