# デプロイって何？

デプロイとは、作ったアプリやWebサイトを、利用者が使える場所へ反映することです。

自分のPCで動いているだけでは、他の人はそのアプリを使えません。

利用者がブラウザでアクセスできるように、サーバーやクラウド上の環境へ反映する必要があります。この反映作業がデプロイです。

```txt
自分のPCで開発
  ↓
変更を確認
  ↓
サーバーやクラウドへ反映
  ↓
利用者がアクセスできる
```

> まとめ: デプロイは、作ったものを利用者が使える場所へ反映することです。

## ローカルで動く状態と公開されている状態

開発中は、自分のPC上でアプリを動かして確認します。

たとえば、次のようなコマンドでローカルサーバーを起動します。

```bash
npm run dev
```

この場合、ブラウザで `http://localhost:5173/` のようなURLを開いて確認します。

これは自分のPC上で動いている状態です。

一方で、利用者がアクセスするWebサイトは、通常 `https://example.com` のようなURLで公開されています。

```txt
ローカル: 自分のPCで確認する状態
公開環境: 利用者がアクセスできる状態
```

ローカルで動いているだけでは、利用者には見えません。利用者が使える状態にするには、デプロイが必要です。

## buildとdeployの違い

デプロイと一緒によく出てくる言葉に、ビルドがあります。

ビルドは、ソースコードを実行・配信しやすい形に変換する作業です。

たとえば、フロントエンド開発では次のようなコマンドを使うことがあります。

```bash
npm run build
```

このコマンドによって、公開用のファイルが作られます。

```txt
ソースコード
  ↓ build
公開用ファイル
```

デプロイは、その公開用ファイルやアプリケーションを、サーバーやクラウド上の環境へ反映する作業です。

```txt
公開用ファイル
  ↓ deploy
利用者がアクセスできる環境
```

つまり、ビルドとデプロイは役割が違います。

```txt
build: 動かしやすい・配信しやすい形に作る
deploy: 利用者がアクセスできる場所へ反映する
```

> ビルドは「公開用に作る」、デプロイは「公開環境へ反映する」と考えると分かりやすいです。

## pushとdeployは違う

Gitを使っていると、`git push` とデプロイを混同しやすいです。

`git push` は、自分のコミットをリモートリポジトリへ送る操作です。

```txt
ローカルリポジトリ
  ↓ git push
GitHubなどのリモートリポジトリ
```

一方で、デプロイはアプリを利用者が使える環境へ反映する操作です。

```txt
GitHubなどのリモートリポジトリ
  ↓ deploy
サーバーやクラウド
  ↓
利用者がアクセス
```

プロジェクトによっては、GitHubにpushすると自動でデプロイされる仕組みがあります。

そのため、見た目上は「pushしたらサイトが更新された」ように見えることがあります。

ただし、正確には、pushとデプロイは別の操作です。pushをきっかけに、デプロイが自動で走っているだけです。

## デプロイ先の例

デプロイ先にはいろいろな種類があります。

たとえば、次のようなものがあります。

- Webサイトを公開するホスティングサービス
- アプリケーションを動かすクラウド環境
- 会社が管理するサーバー
- 静的ファイルを配信するストレージ

FDE人材としては、最初からサービス名をすべて覚える必要はありません。

まずは、デプロイ先は「利用者がアクセスするためにアプリやファイルを置く場所」と考えると分かりやすいです。

## プレビュー環境と本番環境

チーム開発では、いきなり本番環境にデプロイしないことがあります。

まず、確認用の環境にデプロイして、問題がないか確認します。

このような確認用の環境は、プレビュー環境やステージング環境と呼ばれることがあります。

```txt
ローカル環境: 自分のPCで確認する
プレビュー環境: チームや関係者が確認する
本番環境: 実際の利用者が使う
```

たとえば、Pull RequestごとにプレビューURLが発行される仕組みがあります。

```txt
Pull Requestを作る
  ↓
プレビュー環境に自動デプロイ
  ↓
画面を確認
  ↓
問題なければmainへマージ
  ↓
本番環境へデプロイ
```

プレビュー環境があると、コードだけでなく、実際の画面や動作を確認しやすくなります。

## デプロイ後に確認すること

デプロイは、反映して終わりではありません。

デプロイ後には、意図した通りに動いているか確認します。

たとえば、次のような観点があります。

- 画面が表示されるか
- 主要なボタンやフォームが動くか
- ログインや保存などの重要な機能が動くか
- エラーが出ていないか
- 反映したはずの変更が見えているか

特に本番環境では、利用者に影響するため、確認が重要です。

> デプロイは「反映する作業」と「反映後に確認する作業」まで含めて考えると実務に近くなります。

## 問題が起きたら戻すこともある

デプロイ後に問題が見つかることもあります。

たとえば、画面が表示されない、ログインできない、保存できない、といった問題です。

その場合は、修正して再デプロイすることもあれば、前の安定した状態に戻すこともあります。

前の状態に戻すことを、ロールバックと呼ぶことがあります。

```txt
新しい変更をデプロイ
  ↓
問題が見つかる
  ↓
前の安定した状態へ戻す
```

Gitで変更履歴を残しておくと、どの変更を反映したのか、どこから問題が起きたのかを追いやすくなります。

## まず押さえること

最初に押さえるべきことは次の4つです。

- デプロイは、作ったものを利用者が使える場所へ反映すること
- ローカルで動くことと、公開環境で動くことは別
- buildは公開用に作る作業、deployは公開環境へ反映する作業
- pushとdeployは別だが、pushをきっかけに自動デプロイされることがある

> デプロイを理解すると、Git、サーバー、ローカル環境、本番環境の話がつながりやすくなります。

## 理解度チェック

Q1. デプロイの説明として最も近いものはどれですか。

- A. 作ったアプリやWebサイトを、利用者が使える場所へ反映すること
- B. Gitのコミットメッセージを短くすること
- C. ローカルのファイルをすべて削除すること
- D. ブラウザのタブを整理すること

解説: デプロイは、作ったものをサーバーやクラウド上の環境へ反映し、利用者が使える状態にすることです。

Q2. buildとdeployの違いとして最も近いものはどれですか。

- A. buildはGitHubへ送る操作、deployはコミットを作る操作
- B. buildは公開用に作る作業、deployは公開環境へ反映する作業
- C. buildはローカルサーバーを止める操作、deployはエラーを消す操作
- D. buildとdeployは完全に同じ意味

解説: buildはソースコードを配信しやすい形に作る作業で、deployはそれを利用者がアクセスできる環境へ反映する作業です。

Q3. `git push` とデプロイの関係として最も近いものはどれですか。

- A. `git push` は必ず本番環境の画面を直接書き換える操作
- B. デプロイすると必ずGitのブランチが削除される
- C. `git push` とデプロイは別だが、pushをきっかけに自動デプロイされることがある
- D. `git push` すれば確認作業は不要になる

解説: `git push` はリモートリポジトリへコミットを送る操作です。プロジェクトによっては、それをきっかけにデプロイが自動で走ることがあります。

Q4. プレビュー環境の説明として最も近いものはどれですか。

- A. 自分のPCだけで動くlocalhostのこと
- B. Gitのコミット履歴を保存する場所
- C. 利用者が必ず本番利用する唯一の環境
- D. チームや関係者が、本番前に変更内容を確認するための環境

解説: プレビュー環境やステージング環境は、本番に出す前に画面や動作を確認するために使われます。

Q5. ロールバックの説明として最も近いものはどれですか。

- A. 新しい変更に問題が見つかったとき、前の安定した状態へ戻すこと
- B. ローカルサーバーを初めて起動すること
- C. `node_modules` をGitHubに送ること
- D. CSSだけを自動で整形すること

解説: デプロイ後に問題が起きた場合、前の安定した状態に戻すことをロールバックと呼ぶことがあります。

答え:

- Q1: A
- Q2: B
- Q3: C
- Q4: D
- Q5: A
