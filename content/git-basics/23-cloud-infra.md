# クラウド / インフラ

インフラは、アプリケーションを動かすための土台です。

クラウドは、その土台をインターネット経由で利用できるサービスです。

アプリのコードを書くだけでは、利用者がWebサービスを使える状態にはなりません。アプリを動かすサーバー、データを保存するDB、ファイルを置くストレージ、通信を受けるネットワークなどが必要です。

> まとめ: インフラはアプリを支える土台、クラウドはその土台を必要に応じて借りられる仕組みです。

## インフラとは

インフラは、アプリケーションを動かすために必要な周辺の土台です。

Webアプリでは、たとえば次のようなものがインフラに含まれます。

| 要素 | 役割 |
| --- | --- |
| サーバー | アプリを動かす |
| ネットワーク | 利用者や外部サービスと通信する |
| データベース | ユーザー情報や注文情報などを保存する |
| ストレージ | 画像やPDFなどのファイルを保存する |
| DNS | ドメインを接続先へ向ける |
| 監視 / ログ | 問題が起きていないか確認する |
| セキュリティ設定 | 誰がアクセスできるかを制御する |

アプリが動かないときは、コードだけでなく、インフラ側の設定や状態が原因になることもあります。

## クラウドとは

クラウドは、サーバーやDB、ストレージなどをインターネット経由で利用できるサービスです。

以前は、自社で物理サーバーを用意して管理することが多くありました。クラウドを使うと、自分で物理サーバーを買わなくても、必要な分だけインフラを使えます。

```txt
自分たちで物理サーバーを用意する
  ↓
購入・設置・保守が必要

クラウドを使う
  ↓
必要なサーバーやDBをサービスとして利用する
```

代表的なクラウドには、AWS、Google Cloud、Microsoft Azureなどがあります。

## クラウドで使うものの例

クラウドでは、目的に応じてさまざまなサービスを使います。

| 使いたいもの | クラウドでの例 |
| --- | --- |
| アプリを動かす場所 | サーバー、コンテナ実行環境、ホスティング |
| データを保存する場所 | データベース |
| 画像やPDFを保存する場所 | オブジェクトストレージ |
| アクセスを受ける入口 | ロードバランサー、CDN |
| 設定や秘密情報 | 環境変数、シークレット管理 |
| 状態を確認する仕組み | ログ、監視、アラート |

細かいサービス名を最初から覚える必要はありません。まずは「アプリを動かすには、複数の土台が組み合わさっている」と押さえることが大切です。

## Webアプリが動く流れ

クラウド上でWebアプリを動かすときは、ざっくり次のような流れになります。

```txt
利用者
  ↓ URLへアクセス
DNS
  ↓ 接続先へ向ける
クラウド上のアプリ
  ↓ 必要に応じて問い合わせ
DB / ストレージ
```

たとえば、ECサイトで商品一覧を開くとします。

```txt
ブラウザがリクエストを送る
  ↓
アプリが商品データをDBから取得する
  ↓
商品画像をストレージから取得する
  ↓
画面に商品一覧が表示される
```

このように、利用者が見ている画面の裏側では、複数のインフラ要素が連携しています。

## スケールとは

スケールは、利用者やアクセスが増えたときに、処理できる量を増やす考え方です。

たとえば、アクセスが増えると、1台のサーバーだけでは処理しきれなくなることがあります。

その場合、サーバーを増やしたり、より性能の高いサーバーに変えたりします。

| 方法 | 意味 |
| --- | --- |
| スケールアップ | 1台のサーバーをより高性能にする |
| スケールアウト | サーバーの台数を増やす |

クラウドでは、必要に応じてリソースを増やしたり減らしたりしやすい点が特徴です。

## 監視とログ

本番環境では、アプリが動いているか、エラーが出ていないかを確認する必要があります。

そのために、監視やログを使います。

| 仕組み | 見るもの |
| --- | --- |
| ログ | エラー内容、処理の記録、アクセス状況 |
| 監視 | CPU使用率、メモリ使用量、エラー数、レスポンス時間 |
| アラート | 問題が起きたときの通知 |

たとえば、APIが `500 Internal Server Error` を返している場合、アプリのログを見ることで原因を探すことがあります。

## セキュリティ設定

クラウドでは、誰が何にアクセスできるかを適切に設定する必要があります。

たとえば、次のような観点があります。

- 管理画面にアクセスできる人を制限する
- DBに直接アクセスできる範囲を制限する
- APIキーやパスワードを安全に管理する
- 本番環境の設定を誤って公開しない

インフラは「動けばよい」だけではなく、安全に運用できることも重要です。

## まず押さえること

最初に押さえるべきことは次の4つです。

- インフラは、アプリを動かすための土台
- クラウドは、サーバーやDBなどの土台を必要に応じて使える仕組み
- Webアプリは、サーバー、DB、ストレージ、DNSなどが連携して動く
- 本番運用では、監視、ログ、セキュリティ設定も重要

> クラウド / インフラを理解すると、デプロイ後の動作や障害調査の会話がしやすくなります。

## 理解度チェック

Q1. インフラの説明として最も近いものはどれですか。

- A. アプリケーションを動かすための土台
- B. Gitのコミットメッセージだけを保存する場所
- C. HTMLの文字色を変えるための設定
- D. npmパッケージだけを削除するコマンド

解説: インフラは、サーバー、DB、ネットワーク、ストレージなど、アプリを動かすための土台です。

Q2. クラウドの説明として最も近いものはどれですか。

- A. 必ず自社で物理サーバーを購入する仕組み
- B. サーバーやDBなどの土台をインターネット経由で利用できるサービス
- C. ローカルPCのファイル名だけを変える機能
- D. GitHubのプロフィール画像を保存する場所

解説: クラウドを使うと、自分で物理サーバーを用意しなくても、サーバーやDBなどをサービスとして利用できます。

Q3. スケールアウトの説明として最も近いものはどれですか。

- A. 1台のサーバーをより高性能にする
- B. HTTPSの暗号化を無効にする
- C. サーバーの台数を増やす
- D. DNSレコードを必ず削除する

解説: スケールアウトは、処理できる量を増やすためにサーバーの台数を増やす考え方です。

Q4. 本番環境でログや監視を見る理由として最も近いものはどれですか。

- A. アプリのロゴ画像だけを自動生成するため
- B. Gitのブランチ名を短くするため
- C. DBのテーブル名を必ず英語にするため
- D. エラーや負荷など、問題が起きていないか確認するため

解説: ログや監視を見ることで、エラー内容、負荷、レスポンス時間などを確認し、問題の原因を探しやすくなります。

Q5. クラウド / インフラのセキュリティ設定として適切なものはどれですか。

- A. 本番DBを誰でも直接見られるようにする
- B. APIキーやパスワードを安全に管理し、アクセスできる人や範囲を制限する
- C. すべてのログを見ないようにする
- D. 本番環境の設定を公開リポジトリにそのまま置く

解説: クラウドでは、秘密情報の管理やアクセス制御を適切に行い、安全に運用することが重要です。

答え:

- Q1: A
- Q2: B
- Q3: C
- Q4: D
- Q5: B
